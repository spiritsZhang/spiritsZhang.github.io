从零搭建 Hexo 技术博客：一次完整的工程化复盘

发表于2026-06-07|更新于2026-06-10|工程实践
|浏览量:

个人技术博客不只是文章容器，也是一份长期维护的工程作品。本文记录本站从初始化、主题配置到 GitHub Pages 部署的完整过程，并总结其中值得复用的排查方法。

技术选型

为什么选择 Hexo

Hexo 使用 Markdown 管理内容，生成结果是纯静态文件，部署简单，加载速度也比较稳定。对个人技术博客来说，它能让我把精力放在内容和项目复盘上。

为什么选择 Butterfly

Butterfly 提供文章目录、分类、标签、代码高亮和深色模式等能力。主题配置与内容分离，后续升级主题时也更容易保留个人设置。

常见排查顺序

  1. 确认访问的是文章详情页，而不是博客首页。
  2. 确认正文包含 ## 或 ### 标题。
  3. 确认主题配置中的 toc.post 已开启。
  4. 清理旧的生成文件后重新构建。
  5. 检查浏览器缓存和 GitHub Pages 部署结果。

配置与内容分离

我将站点信息放在 Hexo 主配置中，将导航、目录和侧边栏放在 Butterfly 配置中，把文章与页面保存在 source 目录。

这种组织方式的好处是：

  • 配置职责清晰，排查问题更快。
  • 不直接修改主题源码，降低升级成本。
  • 内容文件可以独立进行版本管理。

面向校招的内容规划

项目复盘

每个项目会重点说明问题背景、技术方案、本人职责、关键难点和可验证结果，而不只是罗列技术名词。

技术文章

文章会优先记录真实解决过的问题，包括故障现象、排查过程、方案取舍和最终验证。

学习路径

后续内容将围绕后端基础、数据库、计算机网络、Linux、AI 应用与工程实践持续更新。

后续改进

  • 补充可运行项目及源码说明。
  • 为核心项目增加架构图和演示截图。
  • 完善自动构建与发布流程。
  • 持续整理高质量技术复盘。

博客本身也是一个持续迭代的项目。比起一次性追求复杂效果，我更看重内容是否真实、结构是否清晰，以及每次修改能否被验证。
